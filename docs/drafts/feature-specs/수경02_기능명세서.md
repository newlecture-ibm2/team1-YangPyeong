## 공통 규칙

본 문서 전체에 걸쳐 적용되는 데이터 속성, 용어 규칙, 수급 계산 원리, 검증 정책을 정의한다.

### 데이터 속성 분류

플랫폼에서 다루는 기능과 데이터는 획득 방식과 신뢰도에 따라 크게 3가지로 구분하며, 기능 설계 시 이 기준을 따른다.

1. **공공/API 기반 실데이터** (초기부터 즉시 연동 가능한 실제 데이터)
   - 기상·기후 정보 (기상청 API)
   - 농산물 가격 정보 (KAMIS API)
   - 지역/작물 기본 통계 (통계청)
   - 토양 정보 (흙토람 등 가능한 범위 내)
2. **사용자 입력 데이터** (서비스 내에서 직접 입력·축적받는 핵심 데이터)
   - 재배 의향 등록 (본 플랫폼의 핵심)
   - 농가 프로필 및 보유 시설
   - 과거 재배 이력
3. **더미 데이터** (초기 확보가 어려워 MVP 단계에서 임의 구성하는 데이터)
   - 농업 관련: 씨앗 구매량, 실제 수확량, 작물별 선호도, 로스율
   - 커머스/기타 관련: 농산물 판매/주문/장바구니 기능 데이터, 가게 정보의 일부 상세 속성

### 용어 통일 규칙

> 본 문서에서 사용하는 핵심 용어는 아래와 같이 통일한다. 문서 전체에서 동일한 개념에 대해 다른 표현(예: 재배 등록, 재배 계획)을 사용하지 않는다.

| 한글 용어 | 영문 용어 | 정의 | 비고 |
|----------|----------|------|------|
| **재배 의향** | Cultivation Intention | 농민이 특정 작물을 재배할 계획이 있음을 플랫폼에 등록하는 행위 및 그 데이터 | ❌ "재배 등록", "재배 계획" 사용 금지 |
| **재배 의향 등록** | Intention Registration | 재배 의향을 시스템에 최초로 입력하는 행위 | ❌ "의향 신청" 사용 금지 |
| **수급 밸런스** | Supply-Demand Balance | 특정 작물의 공급 예측량과 수요 예측량의 비율 | ❌ "수급 현황", "수급 비율" 혼용 주의 |
| **밸런스 엔진** | Balance Engine | 수급 데이터를 실시간으로 계산·갱신하는 백엔드 시스템 모듈 | |
| **공급 예측** | Supply Forecast | 재배 의향 데이터 기반으로 산출한 예상 생산량 | |
| **수요 예측** | Demand Forecast | 과거 소비 데이터 + 보정 계수로 산출한 예상 소비량 | |

### 수급 계산 원리

> 본 시스템의 핵심 계산 로직이다. 모든 기능(밸런스 조회, 추천, 대시보드)에서 이 공식을 공통으로 참조한다.

#### 공급 예측 (Supply Forecast)

```
공급 예측(톤) = Σ (재배 의향 면적(㎡) × 단위면적당 수확량(kg/㎡) × (1 − 로스율)) / 1000
```

| 변수 | 데이터 출처 | MVP 전략 |
|------|-----------|----------|
| 재배 의향 면적 | 사용자 직접 입력 (자체 수집) | ✅ 실데이터 |
| 단위면적당 수확량 | 통계청 농업총조사 | ⚠️ 주요 30개 작물 실데이터, 나머지 카테고리 평균 |
| 로스율 | 업계 평균 | 🔴 더미 (작물별 10~20% 기본값) |

#### 수요 예측 (Demand Forecast)

```
수요 예측(톤) = 과거 3년 평균 소비량(톤) × 계절 보정 계수 × 인구 변동 계수
```

| 변수 | 데이터 출처 | MVP 전략 |
|------|-----------|----------|
| 과거 3년 평균 소비량 | 통계청 소비동향 + KAMIS 반출량 | ⚠️ 주요 작물 실데이터 |
| 계절 보정 계수 | 내부 산출 (월별 소비 패턴 분석) | ⚠️ 정적 테이블 |
| 인구 변동 계수 | 통계청 인구동향조사 | ⚠️ 시군구별 정적 데이터 |

#### 수급 비율 및 상태 판정

```
수급 비율(%) = (공급 예측 / 수요 예측) × 100
```

| 수급 비율 | 상태 | 코드 | 의미 |
|----------|------|------|------|
| ≥ 130% | 🔴 과잉 경고 | `SURPLUS_WARN` | 가격 하락 위험, 재배 축소 권고 |
| 110% ~ 129% | 🟠 과잉 주의 | `SURPLUS_CAUTION` | 공급 과다 가능성 |
| 90% ~ 109% | 🟢 적정 | `BALANCED` | 수급 균형 |
| 70% ~ 89% | 🔵 부족 주의 | `SHORTAGE_CAUTION` | 공급 부족 가능성 |
| < 70% | 🟣 부족 경고 | `SHORTAGE_WARN` | 가격 급등 위험, 재배 확대 권고 |

> **임계치 설정:** 위 임계치(130/110/90/70)는 관리자가 FR-AD-004(밸런스 엔진 관리)를 통해 동적으로 조정할 수 있다.

#### 추천 점수 산출

```
종합 점수 = (수급 적합도 × W1) + (환경 적합도 × W2) + (수익성 × W3) + (농가 역량 × W4)
```

| 항목 | 기본 가중치 | 산출 기반 데이터 |
|------|-----------|----------------|
| 수급 밸런스 적합도 | W1 = 35% | 위 수급 비율 기반 점수 변환 |
| 지역 환경 적합도 | W2 = 25% | 기상청 기후 + 흙토람 토양 |
| 예상 수익성 | W3 = 25% | KAMIS 가격 × 수확량 − 생산비 |
| 농가 역량 적합도 | W4 = 15% | 재배 이력 + 시설 + 면적 매칭 |

> **가중치 설정:** 기본 가중치는 관리자가 FR-AD-004를 통해 조정 가능하다. 사용자 `priority` 파라미터로도 ±10% 편향시킬 수 있다.

### Soft Validation 정책

> [!WARNING]
> **아래 항목들은 팀 내 합의가 필요한 사항이다.** 현재 문서에는 초안 기준의 처리 방식이 기술되어 있으나, 개발 착수 전 UX·기획·개발팀 간 최종 합의 후 확정해야 한다.

Soft Validation이란 **입력값이 비즈니스 규칙에 어긋나더라도 등록 자체는 허용하고, 경고(warning)만 표시**하는 검증 방식이다.

## FR-CM-001 | 회원가입

### 기능 설명

신규 사용자가 Farm Balance 플랫폼에 계정을 생성한다. 사용자 역할(농민/지자체)에 따라 수집하는 정보와 승인 절차가 상이하다. 관리자 계정은 내부에서만 생성 가능하다.

### 입력 데이터

| 필드명 | 타입 | 필수 | 제약조건 | 비고 |
|--------|------|------|----------|------|
| `email` | String | ✅ | 이메일 형식, 최대 255자 | 로그인 ID로 사용 |
| `password` | String | ✅ | 최소 8자, 영문+숫자+특수문자 각 1개 이상 | BCrypt 해싱 후 저장 |
| `password_confirm` | String | ✅ | `password`와 일치 | 저장하지 않음 |
| `name` | String | ✅ | 2~20자, 한글/영문 | 실명 |
| `phone` | String | ✅ | 010-XXXX-XXXX 형식 | SMS 인증 필요 |
| `role` | Enum | ✅ | `FARMER`, `LOCAL_GOV` | 관리자는 내부 생성만 가능 |
| `region_sido` | String | ✅ | 시도 코드 (17개) | 드롭다운 선택 |
| `region_sigungu` | String | ✅ | 시군구 코드 | 시도 선택 후 연동 |
| `farm_area` | Decimal | 농민만 ✅ | > 0, 단위: ㎡ | 총 경작 면적 |
| `farm_type` | Enum[] | 농민만 ✅ | `RICE`, `VEGETABLE`, `FRUIT`, `SPECIAL` | 복수 선택 가능 |
| `organization` | String | 지자체만 ✅ | 최대 100자 | 소속 기관명 |
| `position` | String | 지자체만 ✅ | 최대 50자 | 직급/직책 |

### 출력 데이터

| 필드명 | 타입 | 설명 |
|--------|------|------|
| `user_id` | UUID | 시스템 생성 고유 식별자 |
| `status` | Enum | `ACTIVE` (농민) / `PENDING_APPROVAL` (지자체) |
| `created_at` | DateTime | 등록 일시 (KST) |
| `message` | String | 결과 메시지 |

### 처리 로직

```
1. 입력값 유효성 검증 (Validation)
   ├── 이메일 형식 검증 + 중복 확인 (DB 조회)
   ├── 비밀번호 정책 검증 + 확인 일치 여부
   ├── 전화번호 형식 검증
   └── 역할별 필수 필드 완전성 확인

2. SMS 인증 확인
   ├── 인증 요청 시 6자리 코드 발송 (유효시간: 3분)
   └── 사용자 입력 코드와 발송 코드 비교

3. 비밀번호 해싱
   └── BCrypt(password, salt_rounds=12)

4. 사용자 레코드 생성
   ├── user 테이블 INSERT
   ├── user_profile 테이블 INSERT (역할별 프로필)
   └── user_region 테이블 INSERT (지역 매핑)

5. 역할별 후처리
   ├── FARMER  → status = ACTIVE, 즉시 이용 가능
   └── LOCAL_GOV → status = PENDING_APPROVAL, 관리자 승인 대기

6. 가입 완료 알림 발송
   ├── 이메일: 환영 메일 + 서비스 이용 안내
   └── LOCAL_GOV: "승인 대기 중" 안내 메일 추가 발송
```

### 예외 상황

| 코드 | 상황 | HTTP | 처리 |
|------|------|------|------|
| `E-CM-001-01` | 이메일 중복 | 400 | "이미 등록된 이메일입니다." |
| `E-CM-001-02` | 비밀번호 불일치 | 400 | "비밀번호가 일치하지 않습니다." |
| `E-CM-001-03` | 비밀번호 정책 미충족 | 400 | "비밀번호는 8자 이상, 영문·숫자·특수문자를 포함해야 합니다." |
| `E-CM-001-04` | SMS 인증 코드 불일치 | 400 | "인증번호가 올바르지 않습니다." |
| `E-CM-001-05` | SMS 인증 코드 만료 | 410 | "인증번호가 만료되었습니다. 재발송해 주세요." |
| `E-CM-001-06` | SMS 발송 실패 (외부 서비스 장애) | 503 | "인증번호 발송에 실패했습니다. 잠시 후 다시 시도해 주세요." |
| `E-CM-001-07` | 유효하지 않은 지역 코드 | 400 | "유효하지 않은 지역 정보입니다." |

## FR-CM-002 | 로그인 / 로그아웃

### 기능 설명

등록된 사용자가 이메일과 비밀번호로 인증하여 플랫폼에 접속하거나, 세션을 종료한다. JWT(Access Token + Refresh Token) 기반 인증을 사용한다.

### 입력 데이터

**로그인:**

| 필드명 | 타입 | 필수 | 제약조건 |
|--------|------|------|----------|
| `email` | String | ✅ | 이메일 형식 |
| `password` | String | ✅ | 평문 (서버에서 해시 비교) |

**로그아웃:**

| 필드명 | 타입 | 필수 | 제약조건 |
|--------|------|------|----------|
| `access_token` | String | ✅ | Authorization 헤더로 전송 |

### 출력 데이터

**로그인 성공:**

| 필드명 | 타입 | 설명 |
|--------|------|------|
| `access_token` | String | JWT Access Token (유효기간: 1시간) |
| `refresh_token` | String | JWT Refresh Token (유효기간: 14일) |
| `user` | Object | `{ user_id, name, role, region }` |
| `expires_at` | DateTime | Access Token 만료 시각 |

**로그아웃 성공:**

| 필드명 | 타입 | 설명 |
|--------|------|------|
| `message` | String | "로그아웃 되었습니다." |

### 처리 로직

```
[로그인]
1. 이메일로 사용자 조회
   └── 미존재 시 → 에러 (보안상 이메일/PW 구분하지 않음)

2. 계정 상태 확인
   ├── SUSPENDED       → "정지된 계정입니다. 관리자에게 문의하세요."
   ├── PENDING_APPROVAL → "승인 대기 중입니다."
   └── ACTIVE          → 다음 단계 진행

3. 로그인 시도 횟수 확인
   └── 최근 30분 내 5회 이상 실패 → 계정 잠금 (30분)

4. 비밀번호 검증
   └── BCrypt.verify(input_password, stored_hash)

5. 토큰 발급
   ├── Access Token : JWT { user_id, role, region, exp: now+1h }
   ├── Refresh Token: JWT { user_id, exp: now+14d }
   └── Refresh Token → DB 저장 (기존 토큰 무효화)

6. 로그인 이력 기록
   └── login_history INSERT { user_id, ip, user_agent, timestamp }

[로그아웃]
1. Access Token 블랙리스트 등록 (Redis, TTL = 토큰 잔여 유효시간)
2. Refresh Token DB에서 삭제
3. 200 OK 응답
```

## FR-FM-001 | 작물 밸런스 조회

### 기능 설명

농민이 전국 또는 특정 지역의 작물별 수급 밸런스(공급 예측 대비 수요 예측 비율)를 지도 및 차트 형태로 조회한다. 로그인 시 농민의 소재 지역이 기본 필터로 설정된다.

### 입력 데이터

| 필드명 | 타입 | 필수 | 제약조건 | 비고 |
|--------|------|------|----------|------|
| `crop_category` | Enum | ❌ | `GRAIN`, `VEGETABLE`, `FRUIT`, `SPECIAL`, `ALL` | 기본값: `ALL` |
| `crop_code` | String | ❌ | 시스템 작물 코드 | 카테고리 내 세부 작물 선택 |
| `region_sido` | String | ❌ | 시도 코드 | 기본값: 사용자 소재 시도 |
| `region_sigungu` | String | ❌ | 시군구 코드 | 시도 선택 시 활성화 |
| `year` | Integer | ❌ | 2024 ~ 현재연도 | 기본값: 현재연도 |
| `season` | Enum | ❌ | `SPRING`, `FALL`, `ALL` | 기본값: 현재 시즌 자동 판별 |

### 출력 데이터

| 필드명 | 타입 | 설명 |
|--------|------|------|
| `map_data` | Object[] | 지역별 수급 상태 지도 데이터 |
| `map_data[].region_code` | String | 시군구 코드 |
| `map_data[].region_name` | String | 시군구명 |
| `map_data[].balance_status` | Enum | `SURPLUS_WARN`, `SURPLUS_CAUTION`, `BALANCED`, `SHORTAGE_CAUTION`, `SHORTAGE_WARN` |
| `map_data[].supply_ratio` | Decimal | 공급/수요 비율 (%) |
| `map_data[].registered_farms` | Integer | 재배 의향 등록 농가 수 |
| `gauge_data` | Object | 선택 작물의 밸런스 게이지 |
| `gauge_data.crop_name` | String | 작물명 |
| `gauge_data.supply_forecast` | Decimal | 공급 예측량 (톤) |
| `gauge_data.demand_forecast` | Decimal | 수요 예측량 (톤) |
| `gauge_data.balance_ratio` | Decimal | 수급 비율 (%) |
| `gauge_data.balance_status` | Enum | 수급 상태 |
| `gauge_data.yoy_change` | Decimal | 전년 동기 대비 변화율 (%) |
| `summary` | Object | 요약 정보 |
| `summary.total_crops` | Integer | 조회 대상 작물 수 |
| `summary.surplus_count` | Integer | 과잉 상태 작물 수 |
| `summary.shortage_count` | Integer | 부족 상태 작물 수 |

### 처리 로직

```
1. 인증 확인
   └── JWT 토큰 검증, role ∈ {FARMER, LOCAL_GOV, ADMIN}

2. 파라미터 초기화
   ├── region_sido 미입력   → 사용자 프로필의 소재 시도
   ├── crop_category 미입력 → ALL
   ├── year 미입력          → 현재 연도
   └── season 미입력        → 현재 날짜 기준 시즌 판별
       ├── 1월~6월  → SPRING
       └── 7월~12월 → FALL

3. 수급 데이터 조회
   ├── balance_data 테이블에서 조건 매칭 레코드 조회
   ├── JOIN crop_master (작물 정보)
   ├── JOIN region_master (지역 정보)
   └── WHERE: year, season, region, crop_category

4. 밸런스 상태 계산 (레코드별)
   ├── supply_ratio = (supply_forecast / demand_forecast) × 100
   ├── supply_ratio ≥ 130 → SURPLUS_WARN
   ├── supply_ratio ≥ 110 → SURPLUS_CAUTION
   ├── supply_ratio ≥ 90  → BALANCED
   ├── supply_ratio ≥ 70  → SHORTAGE_CAUTION
   └── supply_ratio < 70  → SHORTAGE_WARN

5. 지도 데이터 구성
   ├── 시군구별 대표 수급 상태 집계
   └── GeoJSON 좌표와 매핑

6. 게이지 데이터 구성 (특정 작물 선택 시)
   ├── 해당 지역의 공급/수요 예측량 산출
   ├── 전년 동기 데이터 조회 → YoY 변화율 계산
   └── 재배 의향 등록 농가 수 카운트

7. 요약 데이터 구성
   └── 조회 결과 기준 상태별 작물 수 집계

8. 응답 반환 (200 OK)
```

### 예외 상황

| 코드 | 상황 | HTTP | 처리 |
|------|------|------|------|
| `E-FM-001-01` | 유효하지 않은 작물 코드 | 400 | "존재하지 않는 작물 코드입니다." |
| `E-FM-001-02` | 유효하지 않은 지역 코드 | 400 | "존재하지 않는 지역 코드입니다." |

| `E-FM-001-04` | 수요 예측값이 0 | — | 해당 작물은 `UNKNOWN` 상태로 표시, 서버 로그 기록 |
| `E-FM-001-05` | 데이터 갱신 지연 (최종 갱신 1시간 이상 경과) | 200 | 응답에 `data_freshness: STALE` 플래그 추가 |

## FR-FM-002 | 작물 추천 조회

### 기능 설명

농민의 농가 프로필(소재지, 경작 면적, 보유 시설, 재배 이력)과 현재 수급 밸런스 데이터를 종합 분석하여 최적의 재배 작물 상위 5개를 추천한다.

### 입력 데이터

| 필드명 | 타입 | 필수 | 제약조건 | 비고 |
|--------|------|------|----------|------|
| `user_id` | UUID | ✅ | JWT에서 추출 | 프로필 자동 로드 |
| `farm_area_available` | Decimal | ❌ | > 0, 단위: ㎡ | 미입력 시 프로필상 총 면적 사용 |
| `facility_type` | Enum[] | ❌ | `OPEN_FIELD`, `VINYL_HOUSE`, `GLASS_HOUSE`, `SMART_FARM` | 미입력 시 프로필에서 로드 |
| `exclude_crops` | String[] | ❌ | 작물 코드 배열 | 추천에서 제외할 작물 |
| `priority` | Enum | ❌ | `BALANCE`, `PROFIT`, `EASE` | 가중치 우선순위. 기본값: `BALANCE` |

### 출력 데이터

| 필드명 | 타입 | 설명 |
|--------|------|------|
| `recommendations` | Object[] | 추천 작물 리스트 (최대 5개) |
| `recommendations[].rank` | Integer | 추천 순위 (1~5) |
| `recommendations[].crop_code` | String | 작물 코드 |
| `recommendations[].crop_name` | String | 작물명 |
| `recommendations[].total_score` | Decimal | 종합 추천 점수 (0~100) |
| `recommendations[].scores` | Object | 항목별 세부 점수 |
| `recommendations[].scores.balance_score` | Decimal | 수급 밸런스 적합도 (0~100) |
| `recommendations[].scores.environment_score` | Decimal | 지역 환경 적합도 (0~100) |
| `recommendations[].scores.profit_score` | Decimal | 예상 수익성 (0~100) |
| `recommendations[].scores.capability_score` | Decimal | 농가 역량 적합도 (0~100) |
| `recommendations[].reasons` | String[] | 추천 근거 (최소 3개) |
| `recommendations[].balance_status` | Enum | 해당 작물의 현재 수급 상태 |
| `recommendations[].estimated_revenue` | Object | `{ min, max, unit: "원/㎡" }` |
| `profile_used` | Object | 추천에 사용된 프로필 정보 요약 |
| `generated_at` | DateTime | 추천 생성 시각 |

### 처리 로직

```
1. 사용자 프로필 로드
   ├── user_profile 테이블에서 농가 정보 조회
   ├── 입력된 override 값이 있으면 우선 적용
   └── 프로필 불완전 시 → 필수 정보 입력 요청 안내 반환

2. 후보 작물 풀 구성
   ├── crop_master에서 is_active=true 작물 목록 로드
   ├── exclude_crops에 해당하는 작물 제외
   ├── facility_type으로 재배 가능 작물 필터링
   │   └── 예: OPEN_FIELD → 노지 재배 가능 작물만
   └── 후보 작물 리스트 생성

3. 항목별 점수 산출 (후보 작물 각각에 대해)

   3-1. 수급 밸런스 적합도 (balance_score)
        ├── 해당 지역의 작물별 supply_ratio 조회
        ├── SHORTAGE_WARN    → 100점
        ├── SHORTAGE_CAUTION → 80점
        ├── BALANCED         → 60점
        ├── SURPLUS_CAUTION  → 30점
        └── SURPLUS_WARN     → 10점

   3-2. 지역 환경 적합도 (environment_score)
        ├── 기상청 데이터: 해당 지역 평균 기온, 강수량, 일조 시간
        ├── 작물 적정 조건 테이블(crop_master.climate_conditions)과 매칭
        └── 매칭률(%) × 100 → 환경 점수

   3-3. 예상 수익성 (profit_score)
        ├── 최근 3년 KAMIS 평균 가격 조회
        ├── 예상 수확량 = farm_area_available × 단위면적당 평균 수확량
        ├── 예상 매출 = 예상 수확량 × 3년 평균 가격
        ├── 예상 비용 = farm_area_available × 단위면적당 평균 생산비
        ├── 예상 수익 = 예상 매출 − 예상 비용
        └── 정규화(min-max scaling) → 수익 점수

   3-4. 농가 역량 적합도 (capability_score)
        ├── 사용자 재배 이력 조회 → 해당 작물 재배 경험 유무 (+20점)
        ├── 보유 시설과 작물 필수 시설 매칭 (+40점)
        ├── 경작 면적과 작물 최소 권장 면적 비교 (+40점)
        └── 항목별 가중 합산 → 역량 점수

4. 종합 점수 산출
   ├── 기본 가중치: balance(35%) + environment(25%) + profit(25%) + capability(15%)
   ├── priority 파라미터에 따른 가중치 조정
   │   ├── BALANCE → balance +10%, profit −10%
   │   ├── PROFIT  → profit  +10%, balance −10%
   │   └── EASE    → capability +10%, environment −10%
   └── total_score = Σ (score_i × weight_i)

5. 순위 정렬 및 상위 5개 선택
   └── total_score DESC 정렬 → Top 5 추출

6. 추천 근거 문장 생성 (작물별 3개 이상)
   ├── 템플릿 기반 동적 조합
   │   예: "현재 {지역}에서 공급이 부족한 작물입니다. (수급 점수: {n})"
   │   예: "최근 3년 평균 수익성이 상위권입니다. (수익 점수: {n})"
   └── 상위 점수 항목 기준으로 선택

7. 추천 이력 저장
   └── recommendation_history 테이블 INSERT

8. 응답 반환 (200 OK)
```

### 예외 상황

| 코드 | 상황 | HTTP | 처리 |
|------|------|------|------|
| `E-FM-002-01` | 농가 프로필 미완성 (필수 정보 누락) | 400 | "추천을 위해 농가 프로필을 완성해 주세요." + 누락 필드 목록 반환 |
| `E-FM-002-02` | 후보 작물 0개 (조건 과다 제외) | 200 | 빈 배열 + "현재 조건에 맞는 추천 작물이 없습니다. 조건을 완화해 보세요." |
| `E-FM-002-03` | 수급 데이터 기준일 7일 이상 경과 | 200 | 추천 결과 반환 + `data_warning: "수급 데이터가 최신이 아닐 수 있습니다."` |
| `E-FM-002-04` | KAMIS 가격 데이터 미존재 (신규 작물) | — | 해당 작물 profit_score = 50 (중립값), 추천 근거에 "가격 데이터 부족" 명시 |
| `E-FM-002-05` | 추천 엔진 타임아웃 (2초 초과) | 503 | "추천 처리 중 오류가 발생했습니다. 잠시 후 다시 시도해 주세요." |

## FR-FM-003 | 재배 의향 등록

### 기능 설명

농민이 올해 재배 의향(작물, 재배 면적, 파종 예정일)을 등록한다. 등록 즉시 해당 지역의 수급 밸런스에 반영된다. 한 번에 복수 작물의 재배 의향을 등록할 수 있다.

### 입력 데이터

| 필드명 | 타입 | 필수 | 제약조건 | 비고 |
|--------|------|------|----------|------|
| `user_id` | UUID | ✅ | JWT에서 추출 | |
| `intentions` | Object[] | ✅ | 최소 1개, 최대 10개 | 복수 작물 등록 |
| `intentions[].crop_code` | String | ✅ | 유효한 작물 코드 | crop_master 참조 |
| `intentions[].area` | Decimal | ✅ | > 0, 단위: ㎡ | 해당 작물 재배 예정 면적 |
| `intentions[].sowing_date` | Date | ✅ | 오늘 이후 ~ 12개월 이내 | 파종 예정일 |
| `intentions[].expected_harvest_date` | Date | ❌ | sowing_date 이후 | 미입력 시 자동 계산 |
| `intentions[].memo` | String | ❌ | 최대 200자 | 농민 메모 |

### 출력 데이터

| 필드명 | 타입 | 설명 |
|--------|------|------|
| `intention_id` | UUID | 등록 건(묶음)의 고유 ID |
| `registered_items` | Object[] | 등록된 작물별 상세 |
| `registered_items[].item_id` | UUID | 개별 의향 항목 ID |
| `registered_items[].crop_name` | String | 작물명 |
| `registered_items[].area` | Decimal | 등록 면적 |
| `registered_items[].current_balance` | Enum | 등록 후 해당 지역 수급 상태 |
| `total_registered_area` | Decimal | 이번 등록 총 면적 |
| `remaining_farm_area` | Decimal | 프로필상 총 면적 − 기등록 총 면적 |
| `message` | String | 등록 완료 메시지 |
| `created_at` | DateTime | 등록 일시 |

### 처리 로직

```
1. 인증 및 권한 확인
   └── role = FARMER 확인

2. 입력값 유효성 검증
   ├── crop_code → crop_master 존재 여부 + is_active 확인
   ├── area > 0 검증
   ├── sowing_date가 오늘 이후 + 12개월 이내 확인
   └── expected_harvest_date > sowing_date 확인 (입력 시)

3. 면적 정합성 검증
   ├── 기등록 의향 면적(활성 상태) 합산 조회
   ├── 신규 등록 면적 합산
   ├── 총 등록 면적 = 기등록 + 신규
   └── IF 총 등록 면적 > 프로필상 총 경작 면적 × 1.2
       → 경고 메시지 표시 (등록 자체는 허용 — Soft Validation, 공통 규칙 SV-01 참조)

4. 중복 등록 확인
   ├── 동일 연도 + 동일 작물의 활성 의향 조회
   └── 존재 시 → 409 Conflict 반환

5. 재배 의향 레코드 생성 (트랜잭션)
   ├── cultivation_intention 테이블 INSERT (묶음 헤더)
   ├── cultivation_intention_item 테이블 INSERT (작물별 건)
   ├── status = REGISTERED
   └── expected_harvest_date 미입력 시
       └── sowing_date + crop_master.default_cultivation_days 로 자동 산출

6. ★ 수급 밸런스 갱신 트리거
   ├── 이벤트 발행: INTENTION_REGISTERED { region, crop_code, area }
   ├── 밸런스 엔진이 비동기로 이벤트 수신
   ├── 해당 지역+작물의 supply_forecast 재계산
   │   └── supply_forecast += area × crop_master.yield_per_sqm
   └── balance_data 테이블 UPDATE

7. 등록 후 수급 상태 조회
   └── 갱신된 밸런스에서 등록 작물별 현재 상태를 응답에 포함

8. 알림 발송 (비동기)
   └── "재배 의향이 등록되었습니다. [{작물명}] 현재 수급: [{상태}]"

9. 응답 반환 (201 Created)
```

## FR-FM-004 | 재배 의향 수정

### 기능 설명

농민이 이미 등록한 재배 의향의 면적, 파종일 등을 수정한다. 수정 즉시 수급 밸런스에 차분(delta) 반영된다.

### 입력 데이터

| 필드명 | 타입 | 필수 | 제약조건 | 비고 |
|--------|------|------|----------|------|
| `intention_item_id` | UUID | ✅ | 기존 등록 항목 ID | Path Parameter |
| `area` | Decimal | ❌ | > 0, 단위: ㎡ | 수정할 면적 |
| `sowing_date` | Date | ❌ | 오늘 이후 | 수정할 파종일 |
| `expected_harvest_date` | Date | ❌ | sowing_date 이후 | |
| `memo` | String | ❌ | 최대 200자 | |

### 출력 데이터

| 필드명 | 타입 | 설명 |
|--------|------|------|
| `intention_item_id` | UUID | 수정된 의향 항목 ID |
| `changes` | Object | `{ field: { before, after } }` 변경 내역 |
| `current_balance` | Enum | 수정 후 해당 작물 수급 상태 |
| `updated_at` | DateTime | 수정 일시 |

### 처리 로직

```
1. 인증 및 소유권 확인
   ├── JWT에서 user_id 추출
   └── intention_item의 소유자가 요청자인지 확인

2. 상태 확인
   └── status ∈ {REGISTERED, MODIFIED} 만 수정 가능
       └── CANCELLED → "취소된 의향은 수정할 수 없습니다."

3. 변경 사항 존재 확인
   └── 기존값과 동일하면 → "변경할 내용이 없습니다."

4. 변경 전 데이터 스냅샷 저장
   └── intention_item_history 테이블에 현재 상태 백업

5. 필드 업데이트 (Partial Update)
   ├── 입력된 필드만 선택적 업데이트
   ├── status → MODIFIED
   └── updated_at → NOW()

6. 면적 정합성 재검증 (면적 변경 시)
   └── FR-FM-003과 동일한 면적 정합성 로직 (공통 규칙 SV-02 참조)

7. 수급 밸런스 갱신 트리거
   ├── 이벤트: INTENTION_MODIFIED { region, crop_code, old_area, new_area }
   ├── delta = (new_area − old_area) × yield_per_sqm
   └── balance_data.supply_forecast += delta

8. 응답 반환 (200 OK)
```

## FR-LG-001 | 관할 지역 수급 대시보드

### 기능 설명

지자체 담당자가 본인 관할 지역(시군구 단위)의 작물별 수급 현황을 대시보드 형태로 한눈에 조회한다. 과잉/부족 경고 배지, 주요 작물 Top 10 밸런스, 등록률, 전년 대비 변화율을 포함한다.

| `year` | Integer | ❌ | 2024~ | 기본값: 현재연도 |
| `season` | Enum | ❌ | `SPRING`, `FALL`, `ALL` | 기본값: 현재 시즌 |
| `sort_by` | Enum | ❌ | `SURPLUS_DESC`, `SHORTAGE_DESC`, `FARM_COUNT_DESC` | 기본값: `SURPLUS_DESC` |

### 출력 데이터

| 필드명 | 타입 | 설명 |
|--------|------|------|
| `region` | Object | `{ sido, sigungu, name }` |
| `alerts` | Object[] | 과잉/부족 경고 목록 |
| `alerts[].crop_name` | String | 경고 작물명 |
| `alerts[].alert_type` | Enum | `SURPLUS_WARN`, `SHORTAGE_WARN` |
| `alerts[].supply_ratio` | Decimal | 수급 비율 (%) |
| `alerts[].yoy_change` | Decimal | 전년 대비 변화율 (%) |

| `top_crops[].balance_status` | Enum | 수급 상태 |
| `top_crops[].supply_ratio` | Decimal | 수급 비율 (%) |
| `summary` | Object | 요약 통계 |
| `summary.total_farms` | Integer | 관내 등록 농가 수 |

### 처리 로직

```
1. 인증 확인 (role = LOCAL_GOV)

2. 관할 지역 바인딩
   └── user_profile의 region_sigungu → 데이터 범위 설정

3. 경고 데이터 조회
   ├── balance_data에서 SURPLUS_WARN, SHORTAGE_WARN 작물 필터링

   ├── 작물별 등록 농가 수, 총 면적, 수급 상태 집계
   ├── sort_by 기준 정렬
   └── 상위 10개 추출

5. 요약 통계 계산
   ├── 관내 전체 FARMER 계정 수 카운트
   ├── 해당 연도 의향 등록 완료 농가 수 (DISTINCT user_id)
   ├── registration_rate = registered / total × 100
   └── 상태별 작물 수 집계

```

## FR-LG-003 | 관할 지역 재배 의향 현황 조회

### 기능 설명

지자체 담당자가 관할 지역 내 농가들의 재배 의향 등록 현황을 작물별/읍면동별/시간순으로 집계 조회한다. 개인정보 보호를 위해 개별 농가 상세 정보는 제공하지 않으며, 집계 데이터만 제공한다.

### 입력 데이터

| 필드명 | 타입 | 필수 | 제약조건 | 비고 |
|--------|------|------|----------|------|
| `year` | Integer | ❌ | 기본값: 현재연도 | |
| `crop_code` | String | ❌ | 특정 작물 필터 | 미입력 시 전체 |
| `view_type` | Enum | ❌ | `BY_CROP`, `BY_AREA`, `BY_TIMELINE` | 기본값: `BY_CROP` |
| `page` | Integer | ❌ | ≥ 1 | 기본값: 1 |
| `size` | Integer | ❌ | 1~100 | 기본값: 50 |

### 출력 데이터

**BY_CROP:**

| 필드명 | 타입 | 설명 |
|--------|------|------|
| `items[].crop_name` | String | 작물명 |
| `items[].farm_count` | Integer | 등록 농가 수 |
| `items[].total_area` | Decimal | 등록 총 면적 (㎡) |
| `items[].avg_area_per_farm` | Decimal | 농가당 평균 면적 |
| `items[].yoy_area_change` | Decimal | 전년 대비 면적 변화율 (%) |
| `items[].balance_status` | Enum | 현재 수급 상태 |

**BY_AREA:**

| 필드명 | 타입 | 설명 |
|--------|------|------|
| `items[].area_name` | String | 읍면동명 |
| `items[].farm_count` | Integer | 해당 읍면동 등록 농가 수 |
| `items[].total_area` | Decimal | 총 면적 |
| `items[].top_crop` | String | 최다 등록 작물명 |

**BY_TIMELINE:**

| 필드명 | 타입 | 설명 |
|--------|------|------|
| `timeline[].date` | Date | 일자 |
| `timeline[].cumulative_farms` | Integer | 누적 등록 농가 수 |
| `timeline[].cumulative_area` | Decimal | 누적 등록 면적 |
| `timeline[].daily_new_farms` | Integer | 당일 신규 등록 농가 수 |

### 처리 로직

```
1. 인증 확인 (role = LOCAL_GOV)
2. 관할 지역 기준 데이터 범위 설정
3. view_type에 따른 집계 쿼리 실행
   ├── BY_CROP     → 작물별 GROUP BY + COUNT, SUM, AVG
   ├── BY_AREA     → 읍면동별 GROUP BY + 집계
   └── BY_TIMELINE → 일자별 누적 집계 (Window Function)
4. 전년 동기 데이터 조회 → YoY 변화율 계산
5. 개인정보 보호: 결과에서 user_id, 이름, 전화번호 등 개인 식별 정보 제외
6. 페이지네이션 적용
7. 응답 반환 (200 OK)
```

   ├── 파일 서버 또는 S3 업로드
   └── 24시간 유효 Presigned URL 생성

   └── report_job.status → COMPLETED, file_url 저장

9. 완료 알림 발송

## FR-LG-005 | 관내 농가 공지 발송

```

6. 발송 처리
   ├── 즉시 발송 → 비동기 메시지 큐에 적재
   │   ├── IN_APP: notification 테이블 INSERT (사용자별)

| `data_type` | Enum | ✅ | `BALANCE`, `INTENTION_SUMMARY`, `CROP_LIST` | 데이터 유형 |

   ├── BALANCE          → 관할 지역 작물별 수급 현황 데이터

   ├── XLSX: Apache POI 또는 동등 라이브러리
   └── CSV: 직접 생성, UTF-8 BOM 추가 (한글 호환)

| `sources[].total_records_today` | Integer | 오늘 수집된 레코드 수 |

**수동 연동:**

| 필드명 | 타입 | 설명 |
|--------|------|------|
| `job_id` | UUID | 연동 작업 ID |
| `source` | Enum | 대상 소스 |
| `status` | Enum | `STARTED`, `COMPLETED`, `FAILED` |
| `records_processed` | Integer | 처리된 레코드 수 |
| `records_quarantined` | Integer | 이상값으로 격리된 레코드 수 |
| `errors` | Object[] | 오류 상세 (있을 경우) |
| `duration_ms` | Long | 소요 시간 (ms) |

### 처리 로직

```
[모니터링 조회]
1. 인증 확인 (role = ADMIN)
2. data_sync_log 테이블에서 소스별 최근 연동 이력 조회
3. 소스별 상태 판정
   ├── 마지막 성공 6시간 이내              → HEALTHY
   ├── 마지막 성공 6~24시간 또는 연속 2회 실패 → WARNING
   └── 마지막 성공 24시간 초과 또는 연속 3회 실패 → ERROR
4. 스케줄러에서 다음 연동 예정 시각 조회
5. 응답 반환 (200 OK)

[수동 연동]
1. 인증 확인 (role = ADMIN)

2. 중복 실행 방지
   └── 동일 소스의 STARTED 상태 작업 존재 → 409

3. 연동 작업 등록 + 비동기 실행
   └── data_sync_job INSERT { status: STARTED }

   ├── KAMIS  → 농산물 가격 API → price_data 테이블
   ├── KOSTAT → 농업 통계 API  → statistics_data 테이블
   └── KMA   → 기상 관측 API  → weather_data 테이블

5. 데이터 정합성 검증
   ├── 응답 스키마 검증
   ├── 이상값 탐지
   │   ├── 가격: 전일 대비 50% 이상 변동 → quarantine
   │   ├── 기온: 절대값 > 50℃ → quarantine
   │   └── 기타 도메인별 규칙
   └── quarantine 처리된 레코드는 관리자 수동 확인 대기

6. 정상 데이터 저장
   ├── force=false → 기존 데이터 있으면 SKIP
   └── force=true  → 기존 데이터 UPSERT

7. 작업 완료 처리
   └── data_sync_job.status → COMPLETED 또는 FAILED

8. 밸런스 엔진에 갱신 이벤트 발행
   └── DATA_SOURCE_UPDATED { source, affected_crops }

9. 응답 반환 (200 OK)
```

1. 인증 확인 (role = ADMIN)

2. 입력값 유효성 검증
   ├── 임계치 논리 순서: surplus_warn > surplus_caution > 100 > shortage_caution > shortage_warn
   └── 추천 가중치 합계 = 100 검증

3. 변경 전 설정 백업
   └── engine_config_history INSERT { config_snapshot, admin_id, timestamp }

4. engine_config 테이블 UPDATE

5. 엔진 Hot Reload 이벤트 발행
   └── 밸런스 엔진이 새 설정으로 즉시 전환 (재시작 불필요)

6. 감사 로그 기록
   └── audit_log INSERT { admin_id, action: CONFIG_UPDATED, changes, timestamp }

7. 응답 반환 (200 OK)
```

### 예외 상황

## FR-AD-005 | 시스템 헬스 모니터링

### 기능 설명

관리자가 Farm Balance 시스템 전체의 운영 상태(API 서버, DB, Redis, 밸런스 엔진, 외부 연동)를 실시간으로 모니터링한다.

### 입력 데이터

| 필드명 | 타입 | 필수 | 제약조건 | 비고 |
|--------|------|------|----------|------|
| `period` | Enum | ❌ | `REAL_TIME`, `HOURLY`, `DAILY` | 기본값: `REAL_TIME` |

### 출력 데이터

| 필드명 | 타입 | 설명 |
|--------|------|------|
| `system_status` | Enum | `HEALTHY`, `DEGRADED`, `DOWN` |
| `checked_at` | DateTime | 헬스 체크 시각 |
| `components` | Object[] | 구성 요소별 상태 |
| `components[].name` | String | 구성 요소명 |
| `components[].status` | Enum | `UP`, `DOWN`, `DEGRADED` |
| `components[].latency_ms` | Integer | 응답 시간 (ms) |
| `components[].uptime_percent` | Decimal | 가동률 (%, HOURLY/DAILY 시) |
| `components[].details` | Object | 추가 정보 (메모리, CPU 등) |
| `metrics` | Object | 주요 운영 지표 |
| `metrics.active_users` | Integer | 현재 접속 사용자 수 |
| `metrics.api_requests_per_minute` | Integer | 분당 API 요청 수 |
| `metrics.avg_response_time_ms` | Integer | 평균 API 응답 시간 (ms) |
| `metrics.error_rate_percent` | Decimal | API 오류율 (4xx+5xx %) |
| `metrics.db_connection_pool` | Object | `{ active, idle, max }` |
| `metrics.balance_engine_queue_size` | Integer | 밸런스 엔진 대기 작업 수 |
| `metrics.disk_usage_percent` | Decimal | 디스크 사용률 (%) |

### 처리 로직

```
1. 인증 확인 (role = ADMIN)

2. 구성 요소별 헬스 체크
   ├── API Server    : 자기 인스턴스 상태 + 메모리/CPU
   ├── Database      : Connection PING + 간단 쿼리 응답 시간
   ├── Redis         : PING + 메모리 사용률 INFO 명령
   ├── Balance Engine: 내부 헬스 엔드포인트 호출
   └── External APIs : 최근 data_sync_log 기반 상태 판정

3. 메트릭 수집
   ├── 현재 접속자  : Redis 세션 토큰 카운트
   ├── API 요청/응답 : 메트릭 수집기(Prometheus 등) 또는 로그 집계
   ├── 오류율       : 최근 5분간 4xx+5xx 비율
   ├── DB 커넥션 풀 : HikariCP 등 풀 매니저에서 조회
   └── 디스크        : OS 명령 또는 JMX

4. period별 데이터 범위
   ├── REAL_TIME → 현재 시점 스냅샷
   ├── HOURLY    → 최근 1시간 평균 + 가동률
   └── DAILY     → 최근 24시간 평균 + 가동률

5. 전체 시스템 상태 판정
   ├── 모든 구성 요소 UP → HEALTHY
   ├── 1개 이상 DEGRADED 또는 비핵심 DOWN → DEGRADED
   └── 핵심 구성 요소(DB, API Server) DOWN → DOWN

6. 응답 반환 (200 OK)
```

### 예외 상황

| 코드 | 상황 | HTTP | 처리 |
|------|------|------|------|
| `E-AD-005-01` | DB 연결 실패 | 200 | DB 컴포넌트 status: DOWN, system_status: DOWN |
| `E-AD-005-02` | 메트릭 수집 타임아웃 | 200 | 해당 메트릭 `value: null`, `error: "수집 타임아웃"` |
| `E-AD-005-03` | Redis 연결 실패 | 200 | Redis 컴포넌트 status: DOWN, system_status: DEGRADED |

## FR-AD-006 | 데이터 정합성 검증

### 기능 설명

관리자가 시스템 내 데이터의 정합성을 검증하고, 이상 데이터를 식별하여 조치(자동 수정 또는 수동 확인 요청) 한다.

### 입력 데이터

| 필드명 | 타입 | 필수 | 제약조건 | 비고 |
|--------|------|------|----------|------|
| `check_type` | Enum | ✅ | `INTENTION_AREA`, `BALANCE_SYNC`, `ORPHAN_DATA`, `ALL` | 검증 유형 |
| `auto_fix` | Boolean | ❌ | 기본값: false | true 시 자동 수정 시도 |

### 출력 데이터

| 필드명 | 타입 | 설명 |
|--------|------|------|
| `check_results` | Object[] | 검증 항목별 결과 |
| `check_results[].check_name` | String | 검증 항목명 |
| `check_results[].description` | String | 검증 내용 설명 |
| `check_results[].status` | Enum | `PASS`, `WARN`, `FAIL`, `ERROR` |
| `check_results[].total_checked` | Integer | 검증 대상 건수 |
| `check_results[].issues_found` | Integer | 이상 건수 |
| `check_results[].details` | Object[] | 이상 데이터 상세 (최대 100건) |
| `check_results[].details[].record_id` | String | 이상 레코드 식별자 |
| `check_results[].details[].issue` | String | 이상 내용 설명 |
| `check_results[].details[].severity` | Enum | `WARN`, `FAIL` |
| `check_results[].auto_fixed` | Integer | 자동 수정 건수 (auto_fix=true 시) |
| `executed_at` | DateTime | 실행 시각 |
| `total_duration_ms` | Long | 총 소요 시간 |

### 처리 로직

```
1. 인증 확인 (role = ADMIN)

2. 동시 실행 방지
   └── 기존 진행 중인 검증 존재 시 → 409

3. 검증 유형별 실행

   [INTENTION_AREA] — 의향 면적 정합성 검증
   ├── 각 농가의 활성 의향 등록 총 면적 vs 프로필상 경작 면적
   ├── 총 등록 면적 > 경작 면적 × 1.5 → WARN
   ├── 총 등록 면적 > 경작 면적 × 2.0 → FAIL
   └── auto_fix 불가 (사용자 확인 필요 → 해당 농가에 알림 발송)

   [BALANCE_SYNC] — 밸런스 동기화 검증
   ├── 의향 데이터 기반 공급 예측 재계산
   ├── 재계산 값 vs balance_data.supply_forecast 비교 (지역+작물별)
   ├── 차이율 > 5%  → WARN
   ├── 차이율 > 10% → FAIL
   └── auto_fix=true → balance_data를 재계산 값으로 갱신

   [ORPHAN_DATA] — 고아 데이터 검증
   ├── 탈퇴/삭제된 사용자의 활성 의향 데이터 존재 여부
   ├── 비활성 작물(is_active=false)의 활성 의향 데이터 존재 여부
   └── auto_fix=true → 해당 의향 status를 CANCELLED로 변경

4. 검증 결과 저장
   └── data_integrity_log INSERT { check_type, results, auto_fixed_count, admin_id }

5. 이상 건수 임계치 초과 시 알림
   └── 총 FAIL 건수 > 50 → 관리자 긴급 알림 발송

6. 응답 반환 (200 OK)
```

### 예외 상황

| 코드 | 상황 | HTTP | 처리 |
|------|------|------|------|
| `E-AD-006-01` | 검증 중 DB 타임아웃 (특정 검증 항목) | 200 | 해당 항목 status: ERROR, 나머지 항목은 정상 진행 |
| `E-AD-006-02` | auto_fix 대상 100건 초과 | 200 | auto_fix 중단 + `warning: "영향 범위가 큽니다({n}건). 수동 확인 후 조치해 주세요."` |
| `E-AD-006-03` | 동시 실행 시도 | 409 | "정합성 검증이 이미 진행 중입니다." |
