# 🤖 팜밸런스 Multi-Agent System (MAS) 도입 계획

이 문서는 FarmBalance 프로젝트의 기존 AI 아키텍처를 확장하여, 도메인별 전문 에이전트들이 서로 소통하며 협업하는 **진정한 의미의 멀티 에이전트 시스템(MAS)**을 구축하기 위한 설계와 구현 계획을 정의합니다.

## 1. 아키텍처 비교 분석 (MAS vs Single Core Agent)

강사님의 권장 사항에 따라 최종 채택된 **다중 에이전트 소통 방식(MAS)**과, 대안이었던 **단일 에이전트 방식**의 장단점 비교입니다.

| 비교 항목 | 단일 코어 에이전트 (Single Agent + Tools) | 멀티 에이전트 협업 시스템 (Multi-Agent System) ⭐️ |
| :--- | :--- | :--- |
| **작동 방식** | 1개의 똑똑한 AI가 모든 도메인의 API(Tool)를 직접 호출 | 오케스트레이터가 질문을 분석하고, 전문 에이전트들에게 지시하여 답변 취합 |
| **장점** | 응답 속도(Latency)가 빠름. API 호출 비용이 상대적으로 저렴함. | 도메인별 프롬프트 분리로 **답변의 정확도 및 전문성 극대화**. 코드 확장 및 유지보수가 매우 쉬움. |
| **단점** | 기능이 추가될수록 프롬프트가 길어져 AI가 혼란(환각)을 겪기 쉬움. | 여러 에이전트를 거치므로 응답 시간이 약간 길어질 수 있음. (스트리밍으로 보완 가능) |
| **도입 이유** | 비용/속도 최적화 목적 | **최신 AI 트렌드(LangGraph) 반영**, 각 도메인의 복잡성 격리, 팀 단위 개발 용이성 |

> 💡 **결론**: 팜밸런스는 기능(농장, 수급, 정책 등)이 명확히 구분된 복합 플랫폼이므로, 프롬프트 한계점 극복과 높은 확장성을 제공하는 **MAS(Multi-Agent System) 아키텍처를 채택**합니다.

---

## 2. 아키텍처 개요 및 폴더 구조 (LangGraph 기반)

오케스트레이터(팀장)와 각 도메인을 전담하는 하위 에이전트(직원)들이 LangGraph 위에서 서로 메시지를 주고받으며(Message Passing) 통신합니다.

```text
ai/app/
├── agents/
│   ├── tools/
│   │   ├── farm_tools.py      # 농장 DB(면적, 토양, 날씨) 연동
│   │   ├── balance_tools.py   # 수급 DB(통계, 위기단계) 연동
│   │   ├── policy_tools.py    # 기존 PolicyService 연동
│   │   ├── shop_tools.py      # 직거래 상점(가격, 재고) 연동
│   │   └── community_tools.py # 커뮤니티 RAG 연동
│   ├── farm_agent.py          # 🧑‍🌾 농장 전문 에이전트 (독립된 프롬프트 보유)
│   ├── balance_agent.py       # 📊 수급 통계 전문 에이전트
│   ├── policy_agent.py        # 🏛️ 정책/지원금 전문 에이전트
│   ├── shop_agent.py          # 🛒 상점/유통 전문 에이전트
│   ├── community_agent.py     # 🗣️ 농업 지식/커뮤니티 전문 에이전트
│   └── orchestrator.py        # 🌟 에이전트 간 소통 및 라우팅을 총괄하는 매니저
└── routers/
    └── agent.py               # 오케스트레이터와 연결된 SSE 스트리밍 엔드포인트
```

## 3. 에이전트 역할 및 소통 방식

### 🌟 오케스트레이터 에이전트 (팀장/매니저)
- **역할**: 유저의 질문을 파악해 **어떤 직원(하위 에이전트)에게 물어볼지 결정**합니다. 여러 직원의 대답을 종합해 최종 답변을 유저에게 전달합니다.
- **LLM 전략**: 판단 속도가 매우 빠른 **Groq (Llama 3)**를 사용하여 매니저의 라우팅 지연 시간을 최소화합니다.

### 🧑‍💼 도메인별 하위 에이전트 (전문 직원 5명)
각 에이전트는 자신만의 고유한 시스템 프롬프트(페르소나)와 전용 Tool을 가집니다.
- **Farm Agent**: 농장 상태 파악 담당. "A농장 면적 확인해줘"라는 오케스트레이터의 요청에 응답.
- **Balance Agent**: 거시적 통계 담당. "양평군 감자 수급 비율이 어떻게 돼?"라는 요청에 응답.
- **Policy Agent**: 보조금 및 정책 RAG 담당.
- **Shop / Community Agent**: 각각 유통 가격, 농업 지식(Q&A) RAG 담당.
- **LLM 전략**: 깊은 문서 이해와 데이터 분석이 필요한 하위 에이전트들은 컨텍스트 윈도우가 큰 **Gemini 1.5 Pro/Flash**를 사용하여 정확도를 극대화합니다.

## 4. 인프라 및 배포 전략 (Cloud AI)

효율적인 개발과 안정적인 서비스 운영을 위해 다음과 같은 계층적 인프라 전략을 채택합니다.

- **개발 및 검증 (Dev/Staging)**: 
  - **Groq**: 실시간 대화 흐름의 고속 처리를 위해 사용.
  - **Gemini**: 복잡한 농업 데이터 분석 및 추천 로직 검증에 사용.
- **실 배포 및 운영 (Production)**: 
  - **AWS Bedrock**: 엔터프라이즈급 보안과 안정성이 보장된 AWS 인프라를 통해 에이전트 시스템을 통합 배포합니다. (Claude 3.5 Sonnet 또는 Llama 3 기반 모델 활용)
  - 모든 에이전트의 API 호출은 AWS VPC 내에서 안전하게 관리됩니다.

## 5. 기존 특화 AI 기능과의 공존 및 통합

기존의 `RecommendService`(작물 추천)나 `PolicyService`(공문 생성)는 폐기되지 않습니다.
- **프론트엔드 연동용**: 현재의 `POST /recommend`, `POST /policy` API는 UI 카드나 PDF 다운로드를 위해 기존 그대로 사용합니다.
- **MAS 통합용**: 하위 에이전트(예: Policy Agent)가 오케스트레이터의 지시를 받을 때, 새롭게 계산하는 대신 **기존의 `RecommendService`나 `PolicyService`를 자신들의 Tool로써 재활용**하여 대답을 구성합니다. (로직 중복 방지)

## 5. 구현 단계별 워크플로우

### Phase 1. 무기(Tools) 연동 개발
백엔드 API 호출 함수 및 기존 AI 서비스 연동 함수를 도메인별 `@tool`로 분리하여 작성합니다.

### Phase 2. 도메인 전문 에이전트 생성
각 도메인 에이전트(`farm_agent.py` 등)를 생성합니다. 이때 각 에이전트는 독립된 LLM 인스턴스와 프롬프트를 가지며, 자신의 도메인에 맞는 Tool만 할당받습니다.

### Phase 3. LangGraph 소통 파이프라인 구축 (`orchestrator.py`)
에이전트들이 서로 대화할 수 있도록 LangGraph의 `StateGraph`를 구성합니다.
1. **[Node: Orchestrator]** 유저 질문 분석 → 어떤 에이전트에게 보낼지 라우팅 결정
2. **[Node: Sub-Agent]** 지시를 받은 에이전트(들)가 Tool을 사용해 데이터 조회 및 자체 결론 도출
3. **[Node: Orchestrator]** 에이전트들의 결론을 취합하여 최종 유저 응답(SSE) 스트리밍
