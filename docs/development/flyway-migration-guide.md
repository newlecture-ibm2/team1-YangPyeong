# 📢 [공지] 백엔드 DB 마이그레이션(Flyway) 도입 안내 및 주의사항

팀원 여러분! 더 이상 배포할 때 DB 테이블이 날아가거나 꼬이는 참사를 막기 위해, 드디어 우리 프로젝트에 **안전띠(Flyway)를 장착**했습니다! 🚀

이에 따라 백엔드 데이터베이스 관리 방식이 변경되었으니, 백엔드 개발자분들은 **아래 2가지 사항을 필독**해 주시기 바랍니다.

## 🛠️ 1. 변경된 점: "스프링이 더 이상 테이블을 자동으로 만들어주지 않습니다!"
기존에는 자바 엔티티(`Entity`) 클래스만 수정하면 스프링이 알아서 테이블 컬럼을 추가해 주었죠? 이제 실무 표준 환경에 맞추기 위해 이 기능(`ddl-auto: update`)을 **완전히 껐습니다.**
이제부터 테이블을 새로 만들거나 컬럼을 추가/수정하려면 무조건 **직접 SQL 스크립트를 작성**해 주셔야 합니다.

## ✍️ 2. 앞으로 테이블을 수정하는 방법
자바 엔티티를 수정하셨다면, 백엔드의 `src/main/resources/db/migration` 폴더 안에 SQL 파일을 추가해 주세요. (스프링이 켜질 때 이 폴더의 SQL을 순서대로 자동 실행해 줍니다.)

### [파일명 작성 규칙] - 🔥엄청 중요!
*   **반드시 `V버전__설명.sql` 형태로 만들어야 합니다. (V는 대문자, 언더바는 무조건 2개 `__`)**
*   *좋은 예:* `V5__add_board_table.sql`, `V6__add_nickname_to_users.sql`
*   *나쁜 예:* `V5_board.sql` (언더바 1개면 무시당하고 테이블 안 만들어집니다!)

## 🚨 3. 절대 하면 안 되는 금기사항 (서버 폭발 주의)
**"과거에 한 번이라도 실행되었던(깃허브에 올라갔던) 옛날 V1, V2 SQL 파일은 절대, 네버 수정하지 마세요!"**

Flyway는 한 번 실행한 파일의 지문(Checksum)을 기억합니다. 만약 옛날 스크립트에서 오타를 발견했다고 **과거 파일을 열어서 수정하고 푸시하면, 지문이 달라졌다고 판단해서 서버가 켜지지 않고 폭발**합니다.
만약 과거 테이블의 컬럼을 고치고 싶다면, 옛날 파일을 건드리는 게 아니라 **무조건 `V7__alter_users_table.sql` 이라는 '새로운' 파일을 추가해서 ALTER 쿼리를 작성**하셔야 합니다.

> 궁금한 점이 있거나 새로운 SQL 파일 추가가 낯설다면 언제든 [COMMON/INFRA 담당: 지윤]에게 문의해 주세요! 감사합니다. 😎
