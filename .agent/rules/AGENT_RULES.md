# 🧑‍💻 FarmBalance 개발 에이전트 규칙 (AGENT_RULES)

이 문서는 FarmBalance 프로젝트 개발 시 에이전트가 준수해야 할 표준과 원칙을 정의합니다.

---

## 1. 개발 원칙

- **핵심 유저**: 농업인(고령층 포함), 지자체 담당자, 시스템 관리자.
- **최우선 가치**: UI/UX 접근성(사용 편의성), 정확한 수급 데이터 관리, 간편한 등록 프로세스 구현.

---

## 2. 백엔드 개발 표준 (Java/Spring Boot)

### 2.1 스택 및 아키텍처
- **사용 기술**: Java 21, Spring Boot 3.3.x, Spring Data JPA (Hibernate), Gradle (Groovy).
- **아키텍처 규칙**: **헥사고날 아키텍처 (Hexagonal Architecture)**를 엄격히 준수합니다.
  - `domain`: 핵심 비즈니스 로직 및 엔티티. 프레임워크나 외부 라이브러리 의존성 배제.
  - `application`: 서비스 유스케이스 및 Port(In/Out) 정의.
  - `infrastructure`: 웹 컨트롤러, DB 어댑터, 외부 API 클라이언트 등 세부 구현.

### 2.2 API 설계 및 명명 규칙
- **응답 표준**: 공통 응답 객체(`ApiResponse<T>`)를 사용합니다. 필드명은 `camelCase`, URL 경로는 `kebab-case`를 사용합니다.
- **에러 처리**: `E-[역할]-[기능]-[번호]` 형식의 에러 코드를 정의합니다 (예: `E-FM-SEED-001`).
- **추적**: 모든 요청과 로그에 `X-Correlation-Id`를 포함하여 트래킹이 가능하게 합니다.

---

## 3. 프런트엔드 개발 표준 (Next.js)

### 3.1 기술 스택 및 패턴
- **사용 기술**: Next.js 15 (App Router), React 19, TypeScript 5.
- **스타일링**: **Vanilla CSS** 및 **CSS Modules** (`.module.css`) 사용을 기본으로 합니다.
- **보안 패턴**: BFF(Backend for Frontend) 패턴을 적용합니다.
  - `httpOnly` 쿠키를 사용하여 JWT를 관리하고, 클라이언트 사이드에서 토큰이 직접 노출되지 않도록 합니다.

### 3.2 UI 접근성 (중요 지침)
- **가독성**: 고령 농업인을 배려하여 글꼴 크기는 최소 **16px** 이상을 유지합니다.
- **터치 영역**: 모든 버튼과 클릭 가능한 요소는 최소 **44px** 이상의 터치 영역을 확보합니다.

---

## 4. 작업 프로세스 및 Git 규칙

- **커밋 메시지**: Conventional Commits 규격을 준수합니다 (`feat:`, `fix:`, `docs:`, `chore:` 등).
- **브랜치 관리**: `main` (배포용), `dev` (개발용) 브랜치를 중심으로 작업합니다.
- **환경 구축**: 모든 개발자는 로컬 환경 동기화를 위해 Docker Compose v2를 사용합니다.

---

## 5. 보안 및 에이전트 의무

- **필수 참조 문서**: 에이전트는 모든 작업 수행 전 반드시 `docs/전체_통합.md`와 본 규칙들을 먼저 숙지해야 합니다.
- **개발 맥락 유지**: 항상 "양평군 스마트 파밍 및 수급 균형"이라는 프로젝트의 핵심 목적을 염두에 두고 코드를 작성합니다.
- **언어**: 사용자와의 소통 및 코드 내 주석 처리는 명확한 한국어를 원칙으로 합니다.
